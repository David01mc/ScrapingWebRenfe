Cómo funciona

Un único proceso Python captura, procesa y almacena datos de trenes en tiempo real

1

Captura

Peticiones HTTP cada 30 segundos a los feeds oficiales GTFS-RT de Renfe. Posiciones GPS, retrasos por parada y alertas de servicio.

2

Procesa

Filtrado geográfico por bounding box, cálculo de velocidad y bearing mediante Haversine. Todo en memoria RAM sin tocar la base de datos.

3

Almacena

Flush a Azure SQL cada 4 horas con una única conexión. La base de datos entra en auto-pause entre flushes, minimizando el consumo del tier gratuito.

Cálculo de velocidad y bearing

Renfe no publica velocidad ni rumbo en sus feeds. El sistema los calcula derivándolos de snapshots consecutivos mediante Haversine

El problema

El API GTFS-RT de Renfe incluye solo latitud y longitud. Para cercanías, las coordenadas GPS se actualizan cada 2-10 minutos (no cada 20s como el feed), lo que requiere compensación temporal.

Algoritmo

1Obtener posición anterior de _position_cache (sin BD)
2Si posición no cambió (Δlat < 0.0001): speed = NULL
3Si cambió: buscar en _first_pos_cache cuándo fue la primera vez
4speed = distancia_Haversine / tiempo_real_transcurrido
5Filtrar: < 0.5 km/h → NULL | > 200 km/h cercanías → NULL
python 
def haversine_km(lat1, lon1, lat2, lon2):
    """Distancia en km entre dos coordenadas GPS (fórmula Haversine)."""
    R = 6371  # radio Tierra en km
    dlat = math.radians(lat2 - lat1)
    dlon = math.radians(lon2 - lon1)
    a = (math.sin(dlat / 2) ** 2 +
         math.cos(math.radians(lat1)) * math.cos(math.radians(lat2)) *
         math.sin(dlon / 2) ** 2)
    return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1 - a))

def calculate_bearing(lat1, lon1, lat2, lon2):
    """Rumbo en grados (0-360°) desde el punto 1 al punto 2."""
    lat1_r, lat2_r = math.radians(lat1), math.radians(lat2)
    dlon_r = math.radians(lon2 - lon1)
    x = math.sin(dlon_r) * math.cos(lat2_r)
    y = (math.cos(lat1_r) * math.sin(lat2_r) -
         math.sin(lat1_r) * math.cos(lat2_r) * math.cos(dlon_r))
    return math.degrees(math.atan2(x, y)) % 360

sql
-- Últimas posiciones de Asturias
SELECT TOP 100 *
FROM asturias_vehicle_snapshots
ORDER BY captured_at DESC;

-- Velocidad media por línea en Cádiz
SELECT line,
       ROUND(AVG(speed), 1) AS avg_speed_kmh,
       ROUND(MAX(speed), 1) AS max_speed_kmh,
       COUNT(*) AS registros
FROM cadiz_vehicle_snapshots
WHERE speed IS NOT NULL AND line IS NOT NULL
GROUP BY line
ORDER BY avg_speed_kmh DESC;

-- Retrasos últimas 24h en Asturias
SELECT line,
  ROUND(AVG(CAST(arrival_delay AS FLOAT))/60.0,1) AS avg_delay_min,
  MAX(arrival_delay)/60 AS max_delay_min
FROM asturias_trip_updates
WHERE captured_at >= CONVERT(NVARCHAR,
      DATEADD(hour,-24,GETUTCDATE()),126)
GROUP BY line;

-- Trenes LD en ruta ahora mismo
SELECT cod_comercial, tipo_tren, des_corridor,
       speed, ult_retraso, hora_llegada_sig_est
FROM train_snapshots
WHERE captured_at = (SELECT MAX(captured_at) FROM train_snapshots)
ORDER BY ult_retraso DESC;

Glosario ferroviario & técnico

Definiciones de los términos técnicos que aparecen en este proyecto, ordenados alfabéticamente.

A
GTFS-RT
Alert / Alerta
Mensaje del feed GTFS-RT que notifica incidencias en el servicio: huelgas, obras, cancelaciones o cambios de andén. Se publica en alerts.json.
Azure
Auto-pause
Función de Azure SQL Serverless que detiene automáticamente el cómputo (0 vCores) tras un período de inactividad, ahorrando coste. Al llegar la siguiente conexión, la base de datos se reactiva en segundos.
Ferroviario
AVE
Alta Velocidad Española. Red de trenes de alta velocidad de Renfe que circulan a más de 200 km/h. Comparte infraestructura con Alvia e Intercity en la categoría de Largo Recorrido.
Azure
Azure (Microsoft Azure)
Plataforma de servicios en la nube de Microsoft. Este proyecto usa dos de sus servicios: Azure Virtual Machine (una VM Linux donde corren los scripts de captura) y Azure SQL Database (la base de datos donde se almacenan los datos de los trenes). Ambos forman parte del tier gratuito o de bajo coste para proyectos académicos.
B
Técnico
Batch
Estrategia que acumula datos en memoria RAM durante un período (aquí 4 horas) y los escribe todos de una vez en la base de datos, en lugar de hacerlo registro a registro en cada captura.
Geografía
Bearing (rumbo)
Ángulo de dirección de movimiento medido en grados respecto al norte geográfico (0° = Norte, 90° = Este, 180° = Sur, 270° = Oeste). Se calcula a partir de dos coordenadas GPS consecutivas usando trigonometría esférica.
Geografía
Bounding box (BBox)
Rectángulo geográfico definido por cuatro coordenadas (latitud mínima/máxima, longitud mínima/máxima) que delimita una zona. Se usa para filtrar solo los trenes dentro de esa área geográfica.
C
Ferroviario
Cercanías
Red de trenes de corta distancia operada por Renfe que conecta núcleos urbanos con sus áreas metropolitanas. Las líneas se identifican con C seguido de un número (C1, C2…). Este proyecto captura datos de Asturias y Cádiz.
D
Técnico
DDL
Data Definition Language. Subconjunto de SQL que define la estructura de la base de datos: CREATE TABLE, ALTER TABLE, DROP TABLE. Se ejecuta al arrancar si las tablas no existen aún.
E
GTFS-RT
Entity
Elemento del array entity[] en un feed GTFS-RT. Cada entidad representa un vehículo, un viaje o una alerta, e incluye un identificador único (id) y los datos asociados en tiempo real.
F
GTFS-RT
Feed
Flujo de datos publicado de forma periódica por un servicio. El feed de Renfe se actualiza cada ~20 segundos y contiene posiciones de vehículos, actualizaciones de viajes y alertas de servicio.
Técnico
Flush
Operación de volcado: escribir en la base de datos todos los datos acumulados en memoria y limpiar el buffer. En este proyecto ocurre cada 4 horas, abriendo una única conexión SQL, insertando en lote y cerrando la conexión.
G
GTFS-RT
GTFS
General Transit Feed Specification. Estándar abierto creado por Google para publicar horarios y rutas de transporte público en un formato común. Lo usan miles de operadores de transporte en todo el mundo.
GTFS-RT
GTFS-RT (Realtime)
Extensión de GTFS para datos en tiempo real: posiciones GPS de vehículos, retrasos en paradas y alertas. Renfe lo publica en JSON en gtfsrt.renfe.com actualizándose cada ~20 segundos.
H
Geografía
Haversine
Fórmula matemática que calcula la distancia entre dos puntos sobre la superficie esférica de la Tierra a partir de sus coordenadas GPS. Se usa para calcular la distancia recorrida entre dos capturas y derivar la velocidad del tren.
Técnico
HTTP GET
Método del protocolo HTTP para solicitar datos a un servidor web. El script Python ejecuta un requests.get(url) cada 30 segundos para descargar los feeds JSON de Renfe.
J
Técnico
JSON
JavaScript Object Notation. Formato de texto ligero para intercambiar datos estructurados mediante pares clave–valor y listas. Renfe publica sus feeds GTFS-RT en JSON en lugar del formato binario Protobuf habitual.
L
Ferroviario
Largo Recorrido
Categoría de trenes de Renfe para trayectos de media y larga distancia (AVE, Alvia, Intercity, Talgo…). Sus datos se publican en tiempo-real.largorecorrido.renfe.com cada ~15 segundos.
Técnico
Latencia GPS
Tiempo entre que el tren se mueve y que ese cambio aparece en los datos capturados. En cercanías puede llegar a 2–10 minutos por la baja frecuencia de actualización GPS que publica Renfe.
O
Técnico
ODBC
Open Database Connectivity. Interfaz estándar que permite a aplicaciones conectarse a bases de datos independientemente del fabricante. Este proyecto usa el ODBC Driver 18 for SQL Server de Microsoft en la VM Ubuntu.
P
Ferroviario
Pantógrafo
Dispositivo articulado en el techo del tren que mantiene contacto con la catenaria (el cable eléctrico aéreo) para captar la corriente que alimenta los motores. Elemento distintivo de los trenes eléctricos de alta velocidad.
Técnico
Parallax (efecto)
Técnica visual donde distintas capas de una escena se desplazan a velocidades distintas, creando sensación de profundidad. Cuanto más cercana parece la capa, más rápido se mueve. Usado en la animación del tren de la cabecera.
Técnico
pip
Gestor de paquetes oficial de Python. Instala librerías desde PyPI con pip install nombre. En este proyecto se usa para instalar requests y pyodbc en la VM.
Técnico
pyodbc
Librería Python que permite conectarse y ejecutar consultas SQL en bases de datos mediante ODBC. Es el puente entre los scripts Python y Azure SQL Database en este proyecto.
R
Técnico

Random Access Memory. Memoria de acceso rápido y volátil. Los datos capturados se acumulan en RAM durante 4 horas para minimizar las conexiones a Azure SQL y reducir el consumo de vCore-segundos del tier gratuito.
S
Técnico
Scraping / Scraper
Técnica de extracción automatizada de datos de fuentes externas (APIs o webs) mediante código. Un scraper hace peticiones periódicas, analiza la respuesta y guarda los datos relevantes. Aquí se capturan feeds JSON de Renfe cada 30 segundos.
Azure
Serverless
Modelo cloud donde el cómputo escala automáticamente y se detiene (a 0) cuando no hay actividad. Azure SQL Serverless cobra solo los vCore-segundos realmente consumidos, ideal para cargas de trabajo intermitentes.

Captura puntual del estado de un dato en un instante concreto. Cada 30 segundos se genera un snapshot (fila en la tabla) con la posición, velocidad estimada y timestamp de cada tren capturado.
Técnico

systemd
Sistema de gestión de servicios de Linux que arranca, supervisa y reinicia procesos automáticamente. Los scripts de captura corren como servicios systemd, garantizando que se inicien con la VM y se relancen ante cualquier fallo.
T
GTFS-RT
Trip ID
Identificador único de un viaje programado en el sistema GTFS. Combina número de tren, línea y fecha (p. ej. 3015L23533C1). Permite enlazar la posición GPS de un vehículo con su itinerario de paradas previstas.
GTFS-RT
Trip Update
Mensaje del feed GTFS-RT con el estado en tiempo real de un viaje: retrasos de llegada y salida en cada parada, y cancelaciones parciales o totales. Se publica en trip_updates.json.
V
Azure
vCore
Núcleo virtual (Virtual Core). Unidad de cómputo de Azure SQL. Se cobra en vCore-segundos: vCores asignados × tiempo activo. El tier gratuito incluye 100.000 vCore-s/mes (~28 horas de cómputo a 1 vCore).
Azure
VM (Virtual Machine)
Máquina virtual: ordenador simulado que corre sobre hardware físico en la nube. Este proyecto usa una Standard B1s de Azure (1 vCPU, 1 GB RAM, Ubuntu 22.04) como servidor de captura permanente de datos.
